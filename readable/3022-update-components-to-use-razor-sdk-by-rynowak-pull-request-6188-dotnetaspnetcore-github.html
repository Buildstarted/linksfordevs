Both of these require some discussion.

I expect that the encoding-related issue is not really a bug, ie: wouldn't be user-visible, I'd like avoid the complication of building an HTML encoder into the compiler if possible.

The parser issue with void-elements is already known and is going to take some work for us to resolve without seriously regressing tag helpers.

I don't think either of these is worth blocking upon.

I think I've found the best balance for the testing strategy. The tooling repo has the codegen baseline tests, and this repo has the integration tests that run component code. In theory these integration tests could be written to use components as loose .cshtml and .razor - there are a lot, and it's not broken so I just left them as-is. My personal preference would be destroy all code that uses the existing set of APIs to call into the Razor compiler, but that would be a huge task. I removed all tests that just verify static things about the generated code since those are adequately tested in the tooling repo.
