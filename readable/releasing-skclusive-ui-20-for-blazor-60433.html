Skclusive-UI is Blazor component library based on Material Design.

I am happy to release the version 2.0.0 along with documentation app.

Skclusive-UI Docs App

It is a Blazor app built using Skclusive-UI with lots of demos and code snippets. The docs app is also port of the Material-UI documentation app.

The documentation app is so big that building it took more time and effort than building Skclusive-UI itself. It is worth the effort that could help people trying out the library.

WebAssembly version of the app is live at Github Pages.

AspNetCore Hosted app is live at Azure.

Layout, Markdown and Prism

Also releasing, a component library for responsive layout (Skclusive-Layout), a component library (Skclusive-Prism) to render code highlighting based on Prismjs and a component library (Skclusive-Markdown) to render Markdown content based MarkedNet.

Sample Projects

Sample projects are updated to latest release.

New Skclusive-UI Messenger sample based on Mui Treasury is added.

Skclusive Dashboard is updated with new layout where title, icons and sidebar can be customized easily.

Never thought i would be working on open-source projects until i tried experimenting with Blazor.

As a long time front-end developer, i have advanced personal project setup using ReactJs + TypeScript + MaterialUI + Mobx-State-Tree + React-Router + RxJS + Redux-Observable + Inversify(DI) + Redux-Dev-Tool-Introp with Lerna multi project Monorepo setup (lazy loaded module bundles with module specific mobx-state-tree-store — centralized with time-travel ability)

Being a big fan of C# (started with Silverlight) and developing applications in AspNetCore, when Blazor was introduced as an experimental project I jumped on it with excitement.

But i was bit sceptical because of my relationship with ReactJS (the simplicity and popularity along with vast eco-system of supporting libraries).

Mobx & Mobx-State-Tree

I was confident that there would be lot of Component libraries (open-source and enterprise) for Blazor. But to manage client side states in Blazor, mobx / mobx-state-tree like reactive libraries would be essential. Even in reactjs eco-system mobx is not that popular which should have been the default choice for the advantages it provides.

Hence decided to port mobx in C# and short of achieving my expectation, released those blazor libraries Skclusive.Mobx.Observable and Skclusive.Mobx.StateTree.

Material-UI

Those state management libraries are general purpose and not specific to Blazor component architecture.

Also existing Blazor UI libraries were either using bootstrap or wrapper for material.io components. Honestly not mean to offend here. they are matured and popular of their own. but not of my taste Material-UI.

Hence started porting Material-UI natively in Blazor to understand the Blazor component architecture.

ReactJS to Blazor rewrite is mostly similar but differences are very complex. Especially there is no equivalent for React.cloneElement in Blazor. ReactJS compiles JSX elements as objects (nested child objects) which could be manipulated where as Blazor compiles to component builders.

Styling and Theming

When i started, i could not find CSS-In-JS like libraries in Blazor which is fundamentally used in reactjs based Material-UI. So not wanting to jump on the other track like experimenting with CSS-In-CS(??), instead went ahead using standard CSS with CSS3 custom parameters (variables) which is greater than expected. Changing theme does NOT re-render all the components and it’s associated styles. Only down side is Component’s associates Style components need to be rendered statically at once.

With that said, only few remaining things like bundling, lazy loading of libraries would make Blazor (webassembly) great alternative for JS based front-end frameworks.

But in server rendering Blazor is already an awesome option. I guess which is good enough for most of the apps using AspNetCore.
