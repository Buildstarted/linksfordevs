.NET Core Process APIs not Consistent (or always useful) in all situations · Issue #28990 · dotnet/runtime · GitHub

It seems from discussion in various issues and PR's there is not always consensus about how these concepts (originating in Windows) best map to the world of Unix and containers, given there is also a GC that does measurements of its own, and the OS that by some means of its own chooses when to terminate the process. Attempting to twiddle these values in isolation does not seem likely to help (although I agree PeakWorkingSet64 could surely return WorkingSet64 rather than 0...) and/or may make things worse.

I would like us to understand what the customer scenario is and how they expect to achieve it. For example, is the app trying to adjust the size of its own working set (eg., by releasing cached objects) to avoid getting OOM-killer-ed? If so what API are they using today -- or what API would we recommend they look at, and how should they use it?
