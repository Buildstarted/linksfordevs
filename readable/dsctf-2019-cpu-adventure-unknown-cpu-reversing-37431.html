tl;dr: We reverse-engineered a program written for a completely custom, unknown CPU architecture, without any documentation for the CPU (no emulator, no ISA reference, nothing) in the span of ten hours. Read on to find out how we did it‚Ä¶

This past weekend, I participated in Dragon Sector‚Äôs 2019 Teaser CTF with the CMU PPP team, as a way of de-stressing after a manic CHI 2020 deadline. Dragon Sector is a well-respected Polish team with a history of interesting CTFs, so I wanted to see what they had in store.

After solving ‚Äúummmfpu‚Äù, a challenge that involved reversing bytecode for the Micromega uM-FPU floating-point unit, I decided to tackle the CPU Adventure challenge, which at that point had not been solved by any teams (we would eventually be the only team to solve that challenge).

First Steps

I didn‚Äôt play with the game for too long, figuring that reversing the game.bin file was probably more important. I popped it open in a hex editor, expecting binary ‚Äì imagine my surprise when this is what it looked like:

110011111101000000111100110010001110000011001101000000000000110010011101010000001101001111100001111111001100111000000011...

It‚Äôs literally a binary file ‚Äì a text file containing nothing but ASCII 1s and 0s. We know this is probably machine code for their custom CPU ‚Äì but besides the fact that it has 4 registers, 1 KiB of data memory and 32 KiB of program memory, we know literally nothing else about this CPU. So our first big task is to figure out the unit size of this binary file (e.g. is it 8-bit aligned? or maybe it‚Äôs 12-bit or 18-bit aligned like some ancient architectures?)

To figure out the alignment of an unknown file, I turn to an age-old trick ‚Äì resizing a text window until the line-wrap length matches the file alignment. This works wonders for things like repeating-XOR ciphertext, unknown (uncompressed) file formats, and code from an unknown CPU:

So, from this quick test I figured out that the unit size of this file must divide 20 (the width of the window at alignment). To nail down the exact unit size, I wrote a quick script to look for long repeated strings (figuring that any code would have repeated stereotypical sequences). The longest repeated string was the following 425-bit block, which appeared at positions 43625 and 44510:

We looked for 5-bit punch card encodings, and quickly settled on the historical Baudot code encoding. Indeed, when we used an online decoder to decode some snippets, we got readable text!

When we tried to decode the whole file using Baudot code, we got gibberish for the first 20,000 bits or so, after which we got completely readable text. This suggested that the first part of the file corresponded to a ‚Äúcode‚Äù section, which was followed by a ‚Äúdata‚Äù section containing constant strings. We guessed that the machine probably used Baudot code for I/O, which is why it also stored constant strings in memory using Baudot encoding. To make the code segment more readable, I decided to encode it using a base-32 encoding, similar to hexadecimal encoding but extended to the alphabet 0-9a-v. For simplicity, in the following sections I will refer to the five-bit units as ‚Äúbytes‚Äù. We internally had a bunch other names for these things ‚Äì I called them kibbles, and Zach called them hecs.

Reversing an Unknown CPU Architecture

Well, now we get to the hard part ‚Äì reversing the 4000 bytes of code which are run on a completely unknown, custom CPU architecture. From the code, it‚Äôs pretty evident that it must be a variable-length instruction set, because there‚Äôs no obvious repeating pattern that holds throughout. I spent several hours on this, assisted later by my teammate Zachary Wade (@zwad3). My first order of business was to start looking for repetitive bits of code, under the assumption that they would use a small number of instructions frequently. I started splitting the code up into shorter, repetitive sequences that would be more amenable to analysis. I wish I could say that I had a rigorous process, but it was very much the following fuzzy algorithm:

  • Scan through the code and see if there‚Äôs something that shows up a lot
  • Run a find-replace to stick a newline near that repeat
  • Examine the similarities/differences between the resulting split lines
  • Repeat this process for about an hour‚Ä¶

For example, one pattern I found was ‚Äú0f0.f‚Äù where ‚Äú.‚Äù represented an unknown character. I split on that to get the following:

This is very useful! From the second and third lines, we can see that we have ‚Äú‚Ä¶p5f3r9c‚Ä¶‚Äù and ‚Äú‚Ä¶p5f39c‚Ä¶‚Äù, suggesting that ‚Äúr‚Äù is a one-byte opcode, which means that ‚Äú‚Ä¶5f3‚Äù is the end of one opcode and ‚Äú9c..‚Äù is the start of another. In the last two lines, we have ‚Äúp5f3r1c‚Ä¶‚Äù which means that ‚Äú1c..‚Äù is another opcode, and ‚Äúp3‚Ä¶‚Äù is yet another opcode.

I inferred that ‚Äúp‚Äù and ‚Äúq‚Äù were instructions with three operand bytes, ‚Äúf0‚Äù, ‚Äúf1‚Äù and ‚Äúf2‚Äù were instructions with one operand, and ‚Äú9c‚Äù was an instruction with two operands. I didn‚Äôt know what any of the instructions actually were, though.

When I went to catalog all of the ‚Äúp‚Äù instructions I had extracted, I discovered that by far the most common ‚Äúp‚Äù instruction was ‚Äúp5f3‚Äù. Furthermore, upon looking at all the places it appeared, I found that it was always preceded by ‚Äúf0‚Äù, ‚Äúf1‚Äù and ‚Äúf2‚Äù instructions. Looking at all the ‚Äúf0‚Äù, ‚Äúf1‚Äù and ‚Äúf2‚Äù operands, I noticed that the f2 operands were always in the range of 4-8. Remembering that the CPU had 32 KiB of program memory ‚Äì which requires 15 bits to address ‚Äì I made a guess that ‚Äúf0‚Äù, ‚Äúf1‚Äù and ‚Äúf2‚Äù were loading some kind of address, with f2 as the high byte. When I pieced together some of the addresses, I found they pointed exactly at the start of constant strings in the data ‚Äì I had just found the ‚Äúprint‚Äù function!! This immediately implied that ‚Äúp5f3‚Äù was actually some kind of print string or call instruction; given the three byte operand, a ‚Äúcall‚Äù was most likely. Looking again at all the ‚Äúp‚Äù instructions, I realized that the three operand bytes represented a program address in little-endian order ‚Äì that is, the last operand byte was the most significant address byte.

This was a huge breakthrough! We had figured out our first instruction. Seeing ‚Äúf0‚Äù and ‚Äúf1‚Äù used in some other places, I guessed that perhaps they weren‚Äôt loading addresses ‚Äì instead, maybe they were loading one of the four registers (f0 loading register 0, for example) with 5-bit immediate constants. This would make sense for p5f3 ‚Äì it was loading three register arguments for the function 3f5 (‚Äúprint_string‚Äù).

I started writing a disassembler, which recognized the ‚Äúprint‚Äù idiom (f0x, f1x, f2x, p5f3), placing the printed string inline into the disassembly. Due to the sheer number of strings in the program, this quickly rendered the disassembly very readable, and it was easy to discern where the functional blocks were (full disassembly here):

Just from this little snippet of code, I was able to guess a few things: the ‚Äúq0‚Äù instruction must represent some kind of conditional branch (because it‚Äôs used to skip printing invalid directions in the 1v function), and the instructions ‚Äú9c11‚Äù, ‚Äú9c21‚Äù, ‚Äú9c41‚Äù, ‚Äú9c81‚Äù should represent some kind of AND instruction ‚Äì checking bits set to see if those directions are allowed (the use of ‚Äú1‚Äù, ‚Äú2‚Äù, ‚Äú4‚Äù and ‚Äú8‚Äù in these instructions is quite telling).

For the next two hours, me and Zachary Wade (@zwad3) worked through the various instructions, making and refining our guesses as to what the instructions did. The presence of many readable print statements made our job a lot easier. We decided to each write our own disassembler so we could explore instructions at our own pace and just share our findings back and forth.

Reversing the Code

After a few hours, we were starting to make great progress on disassembly. By looking at the code that worked with the user‚Äôs inventory (specifically, the ‚Äúdrink‚Äù function and every handler associated with it), we found memory store and load instructions (recall that there is 1 KiB of data memory attached to the CPU). We then worked out that certain arithmetic/logic (ALU) instructions took memory operands (e.g. ‚Äú9c41‚Äù actually meant ‚ÄúAND the value at data address 1 with the immediate 4‚Äù). From there, we could reconstruct the variables that lived in data memory ‚Äì for example, [0] held the ID of the NPC at your current location, and [6,7] contained your current health (low 5 bits in [6], high 5 bits in [7]). At this point, I broke away from reversing instructions to start annotating my disassembly and reverse engineering the program itself. You can see my funny notation for 5-bit values (‚Äú0y‚Ä¶‚Äù, a play on ‚Äú0x‚Äù) below:

_start:
    call init

    call print_menu

    brz 2k
    print 6ds # '\x0e- \x0fGO \x0e(\x0fS\x0e)\x0fOUTH\r\n\x00'

There were still lots of unknown opcodes ‚Äì for example, while we figured out that ‚Äúqa‚Äù was a conditional branch-on-zero (brz), we did not figure out what ‚Äúqc‚Äù was (represented as br<c> above). But, it was enough to start figuring out the program logic.

The game lets you basically walk around an 8√ó8 map, in which NPCs (dragons, ‚Äúred bulls‚Äù and people) are randomly placed. You can fight any NPC (even Valis, despite the lack of a menu option to do so). During a fight, you can attack your opponent, dealing random amounts of damage or missing, and your opponent then attacks you, again dealing random damage or missing. Alternatively, you can choose to shield, which causes the opponent‚Äôs attack to either miss or hit your shield and do no damage. Finally, you can cheat, which sets your health to 1000, but also sets a hidden variable (‚Äúcheated‚Äù, address 10) to 1. If you successfully kill your opponent, they drop an item ‚Äì usually, a bottle of alcohol of some kind (clearly, not a kid-friendly game).

Valis, the main NPC who you have to get a flag from, has a state machine in which he asks you for several items ‚Äì a bunch of red bull drinks (obtained by defeating red bull enemies, obviously), various mixed drinks (e.g. a gin and tonic, obtained by defeating a blue dragon and a gray dragon and mixing their drops), and a power strip, obtained by either defeating or helping the other human NPC in the game (Redford). If you finish his long series of requests, he gives you a flag ‚Äì but only if the ‚Äúcheated‚Äù variable isn‚Äôt set. So, our goal is to beat the game without cheating. Since you start with only 100 HP, the same as every enemy, if you play normally it will be impossible to beat every single enemy (you have to beat about 20 to get all items needed). You need to somehow rig the RNG so your opponent always misses.

Random number generation is provided by a function which appears to be some kind of PRNG (address 37a), but it uses unique instructions that aren‚Äôt used anywhere else, so we could not reverse it. However, we noted that it loads its state vector from three memory locations [11], [12] and [13], meaning that its full state is only 15 bits in size. This means that the RNG must have a short period ‚Äì no more than 2^15 = 32768 in length.

Jay Bosamiya (@jay_f0xtr0t) and Matthew Savage (@thebluepichu) implemented the exploit while I was still attempting (futilely) to reverse the RNG implementation. By simply sending the command ‚Äúshield‚Äù 100,000 times in a row, we were able to get a sequence of enemy ‚Äúhits‚Äù and ‚Äúmisses‚Äù which corresponded to the bits output by the RNG. We confirmed that this sequence repeated with a period of 32767. Thus, we were able to build our master exploit ‚Äì at the first enemy we encountered, we shielded 40 times to recover a sequence of hits and misses, searched for the sequence in the big periodic sequence, and then figured out when to shield and when to attack such that the enemy always missed. We simply then walked around the map in murderhobo mode, killing everything and taking their loot. Finally, we returned to Valis and asked nicely for the flag, which we received:

DrgnS{m4kin9-v4lis-happy-w1th-n4t1ve-b4ud0t-cpu}

Whew! What an adventure, indeed. I still can‚Äôt quite believe that we went from a binary string and zero documentation on the CPU to two nearly-complete disassemblers and nice, clean disassembly code in less than 10 hours!
