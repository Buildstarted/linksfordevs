This project aims to:

  • Identify examples that will benefit from running in the interactive try.dot.net environment, which runs C# examples interactively and allows users to modify code.
  • Identify issues that arise when examples are run interactively by using try.dot.net.

The project consists of the following categories:

  • Articles for evaluation. A list of articles that are candidates for try.dot.net. Community members are welcome to open issues or add items with links to articles they believe could benefit from interactivity.
  • Articles made interactive. Issues that were evaluated and found to benefit from interactivity and that now have interactive examples.
  • Issues with interactive examples. This column includes issues that have arisen in running code interactively in the try.dot.net environment. In these cases, try.dot.net produces a result different from that on a user's local system. The issues themselves should be opened in the dotnet/try repo.
  • Resolved issues. This column includes issues with interactive examples that have been resolved.

There are several related goals to this project:

  1. Minimize redundant information. Much of the language reference is duplicated in the spec. They have drifted over time.
  2. Put related features on the same page. That helps readers find alternatives to the syntax they were searching.
  3. Optimize for search on unfamiliar tokens. Readers search when they see unfamiliar syntax. We want them to quickly find what those tokens mean, and learn more quickly.
  4. Consolidate the amount of content.

This project will make those changes in several steps:

  1. Publish the language spec proposals for features released after C# 6.0 (draft spec available)
  2. Create a new "Abridged Language Reference" section that consolidates much of the existing content and refers readers interested in more details to one or more sections of the spec or proposals.
  3. For each sub-section in the "Abridged Language Reference" do the following:
    1. Create the new articles
    2. Search for existing issues on content being migrated and consolidated. This is a good opportunity to fix those.
    3. Redirect existing content to the new articles.
    4. Remove old articles.
    5. Update all links in the .NET Content

The sub-sections are:

  • builtin types
  • operators
    • primary operators
    • unary operators
    • arithmetic operators
    • relational and equality operators
    • type testing and assignment
    • logical operators
    • other operators and special tokens
  • Defining types, members and variables
    • declaring types
    • declaring members
    • controlling access
  • Statements
    • selection and jump statements
    • iteration statements
    • exception statements
    • other statements
  • LINQ query expressions
  • Conversions
  • Attributes
  • Unsafe code
  • Documentation comments
  • Compiler options

This project lists outstanding tasks and work items for which we would appreciate community contributions. The items are grouped into three categories. Depending on your interests, your level of familiarity with GitHub, and your level of comfort with creating independent content, you can choose from the following three categories:

  • Maintenance. These work items involve such things as correcting broken links, correcting spelling or stylistic errors, or modifying the formatting of content to improve the overall appearance of the documentation on docs.microsoft.com.

  • Content updates. Parts of our massive documentation set have become stale and are in need of updating. Some topics don't do a good job of addressing the issues that developers face when using a technology and require revision and refocusing. Migration to a new publishing platform has created duplication and, in some cases, triplication of largely identical topics that needs to be addressed through a process of consolidation.

  • Authoring new content. Every documentation set has gaps. We welcome community contributions to address those gaps. We've identified some of the topics and topic areas for which we would like to see content, but the list is not exhaustive. You may want to contribute a topic in a technology area about which you feel passionate that isn't on the list.

All of these work items are available. Two additional columns contain items for which work either has begun or has been completed.
