To be perfectly honest this is a gut feel about the future more than anything, as the API is still too new for me to have sufficient evidence that people are actively misusing it today. In an ideal world we could rely solely on developer documentation, but being developers ourselves we know that people rarely read documentation. :)

Furthermore, Karel showed me offline that this API has already shipped. So instead of changing the signature of the existing Rent method, the proposal would be to add an overload that also accepts a bool zeroFill parameter. (Or per Morgan's suggestion have an Unsafe-named method.)

To recap: my assumptions are (a) that this API will turn out to be a pit of failure because developers will use it for something it's not intended to be used for, and (b) we don't have a good way of communicating to developers the right way of using this API and ensuring that the audience sees it. If these assumptions are incorrect then there's no real need for this work item.
