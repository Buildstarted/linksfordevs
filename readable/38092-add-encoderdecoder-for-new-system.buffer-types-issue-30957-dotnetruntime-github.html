Add Encoder/Decoder for new System.Buffer types · Issue #30957 · dotnet/runtime · GitHub

The reason the maxCharsPerByte argument doesn't make sense is that it's sometimes impossible to know. For example, say that I perform a "final flush" on an Decoder instance, passing in an empty input:

decoder.Convert(ReadOnlySequence<byte>.Empty, destCharSpan, flush: true, ...);

If the current Decoder instance has internal stored state, it's possible that it could write a non-zero number of characters to the destination span, even with a zero-length input. What should the caller pass for maxCharsPerByte in this situation? No number when multiplied by zero will produce an answer that's large enough to hold the output. It's the same problem as division by zero.

If you wanted to suggest a new API Decoder.GetMaxCharCount(int byteCount, bool flush) that would serve as an O(1) "worst case" equivalent to the existing Decoder.GetCharCount APIs, feel free to open a new tracking issue. It'd probably need to follow a try-pattern instead since otherwise we'd be adding a new abstract method onto an abstract base type, but you get the gist. :)
