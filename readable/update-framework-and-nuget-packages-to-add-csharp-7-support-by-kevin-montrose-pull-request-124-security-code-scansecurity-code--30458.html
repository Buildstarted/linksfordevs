Background

I'm a developer working at Stack Overflow, and I (along with some others) am evaluating some .NET static code analysis tools to see if we can integrate any of them into our development pipeline.

Security Code Scan stood out because of how easy it is to integrate (just a NuGet package) and how simple configuration is. Bonus point for being open source.

Change

We keep our code bases pretty close to the cutting edge, but Security Code Scan doesn't support C# 7.

This updates all projects and Roslyn dependencies up to the point where Visual Studio 2019 and CSC will load all the analyzers and the analyzers understand C# 7. This turns out to be .NETFramework 4.7.2 and Roslyn 3.0.0, determined through some arduous DLL hell trial-and-error.

Also fixes at least one bug that C# 7 introduced.

I've confirmed that a NuGet build of this branch can be successfully integrated into a number of our different codebases. I haven't yet got all tests passing, but most of them do and I'll get behavior matching master before too long.

Question

Before doing too much more work, I wanted to ask: is there any interest in accepting these changes upstream?.

I noticed issue #38 , so I'm hoping it will be. While I cannot guarantee we'll ultimately decide to use Security Code Scan, I can say I'm pretty impressed with it regardless.

If there's interest in accepting these changes, I'll finish the work and turn this into a proper PR in the coming days.
